## 二、面試官三大提問 → 書審該預先回答的內容

### Q1：「你是資工為什麼申請資管？」

**現況問題：** 自傳（3-2）已寫得不錯，但**埋得太深**——要到第 7 頁才看到。

**優化策略：** 在履歷表的開場 5 點摘要（目前第 4 頁）中，把第 5 點改寫得更鋒利。

**修改前：**
> 申請雲科資管二技目的：以系統化的管理理論補足「技術之外」的缺口，朝資訊技術與組織管理整合方向發展

**修改後：**
> **資工→資管的選擇背景**：三段業界實務（DFI 獨立交付、校園 PM、樂聯一人除錯 ERP）讓我確認——資工訓練我「**做得出東西**」，但企業真正缺的是「**做對的東西**」的能力。我選擇雲科資管，是希望以管理理論補上需求分析、專案管理、組織診斷三塊技術之外的核心缺口。

**強化點：** 一句話講完「為什麼來」「憑什麼來」「來了要做什麼」。

---

### Q2：「印象最深的專題你扮演什麼角色？」

**現況問題：** 各專案的角色定位描述太模糊（「我以第一作者身份參與」太被動）。

**優化策略：** 在每個專案的「動機/起點」段落前，新增明確的「**我的角色**」標籤。

#### 範例改寫（TWELF/FC 2025 章節）

```
▸ 我的角色
第一作者 ｜ 系統架構設計 ｜ 資料增強策略主導 ｜ 論文撰寫

從問題定義、模型選型（YOLOv7 + CNN-OCR 雙層架構決策）、
資料增強策略（2,976 → 23,806 筆擴增邏輯設計）、
客製化匹配演算法（歐氏距離 + 邊界框排序），
到論文初稿撰寫與兩場研討會口頭報告——全部親自主導。
團隊另三位成員主要協助資料標註與部分實驗執行。
```

#### 範例改寫（Grade AI 章節）

```
▸ 我的角色
系統架構師 ｜ 全端開發主導 ｜ AI Pipeline 整合

將研討會的演算法核心，獨立設計為 Next.js + FastAPI + MySQL 的
三層架構；非同步推論流程（YOLO → CNN-OCR → 匹配）由我設計
狀態回饋機制；前端教師操作介面亦由我規劃三階段拆解。
```

#### 範例改寫（DFI 章節）

```
▸ 我的角色
主導開發 ｜ 需求訪談 ｜ 系統架構設計 ｜ 系統文件撰寫主導

主導整套 PCB 元件自動化製圖工具的開發。負責需求訪談、
報價、時程協商、系統設計、卷積文字密度演算法設計、
Python 實作，並**主導系統文件的撰寫與交付**——
從演算法原理說明、API 介面文件到操作手冊，
完整建立可供後續工程師接手維護的技術知識庫。

過程中與公司內部工程師、產品端進行多次需求對齊與
驗收溝通，確保技術產出符合實際生產線需求。
```

**為什麼這樣改：**
- 評審 / 面試官想知道「**團隊裡誰真的做了核心**」
- 用標籤式的「角色名稱」直接打中關鍵字（架構師 / 主導 / 文件）
- DFI 強調「文件主導」呼應雲科資管「**軟體品質管理**」「**知識管理**」課程

## 三、樂聯段落徹底改寫（一人除錯 + 硬軟體建議）

這是這次調整裡**改幅最大**的一塊。

### 修改前的問題
- 「除錯協作」聽起來像低階任務、像團隊成員之一
- 沒有展現「一個人面對 15 年舊系統」的獨立判斷力
- 沒有點出「硬體升級建議」這個增值角色

### 修改後完整段落

```
7. 樂聯工業 一人除錯 15 年舊系統 + 硬軟體建議

[標籤] 一人除錯 ｜ 15 年舊系統 ｜ 硬體升級建議 ｜ 軟體重構建議

▸ 我的角色
獨立除錯 ｜ 系統健檢 ｜ 硬軟體升級建議提供者

2025 年，我受邀協助樂聯工業處理一套已運作 15 年的內部系統。
不同於團隊協作的開發專案，這次是**一個人進入現場**，
獨立完成除錯工作，並向公司提供硬體升級與軟體優化的建議。

▸ 進入現場：一個人面對 15 年的程式碼
這套系統無原始開發者、無文件、跨多代技術棧（PHP 4 → 7 混用）。
我必須獨立完成：
1. 透過閱讀 legacy code 還原業務邏輯
2. 與作業員、工程師訪談，補齊文件缺口
3. 在不影響日常運作的前提下定位錯誤與效能瓶頸

▸ 從除錯者升級為建議者：給出軟硬體升級建議
單純「修好 bug」只解決症狀。我額外向公司提出**升級建議**，
涵蓋兩個層次：

· **硬體面**：伺服器規格瓶頸分析、儲存設備升級（SSD vs HDD）
  的成本效益建議

· **軟體面**：哪些模組可修補、哪些建議重構、哪些已過時應淘汰

這讓我從「執行者」轉變為「**建議者**」——必須回答的不只是
「能不能修好」，還包含「**值不值得修**」與「**怎麼修比較划算**」。

▸ 對應雲科資管的學習動機
這次經驗讓我意識到：**技術人的真正價值，是把技術可行性翻譯成
商業決策**。同時也讓我看到「技術債背後是組織」——20 年的程式
碼裡，嵌入的不只是 bug，更是組織的溝通模式、決策慣性與技術
採用邏輯。

這正是雲科資管「**策略資訊管理**」「**組織診斷與更新**」
「**軟體品質管理**」想訓練的核心，也是我從工程師轉向整合型
人才的關鍵起點。
```

### 修改前後對比

| 修改前 | 修改後 |
|--------|--------|
| 「系統除錯協作」 | 「一人除錯 + 硬軟體建議」 |
| 「跨角色溝通」（半學徒） | 「獨立判斷」（半主導） |
| 個人技能展示（讀 code 能力） | 建議價值展示（決策建議能力） |
| 對應「軟體品質管理」單一課程 | 對應 3 門核心課程 |

## 四、DFI 段落調整重點

### 4-1 移除「無人協作」描述

**修改前：**
> 從零到一，無人協作。需求訪談、報價、時程協商、系統設計...

**修改後：**
> 主導整套 PCB 元件自動化製圖工具的開發。從需求訪談、報價、時程協商、系統設計、演算法設計、Python 實作，到**主導系統文件的撰寫與交付**——所有核心技術決策由我負責，並與公司內部工程師、產品端進行多次需求對齊與驗收溝通。

**理由：**
- 不再過度強調「孤狼」形象（容易讓面試官覺得不會團隊合作）
- 改成「**主導 + 與公司端對齊**」更貼近實際情況
- 同時保留「**所有核心技術決策由我負責**」的關鍵主導感

### 4-2 移除「客戶簡報」並強化「系統文件主導」

**修改前的暗示：** 文中暗示「年會評比第四名」是我做的簡報。

**修改後：**
> 工具正式交付後，將工程師原本數小時的手動作業縮短至數秒，並獲得公司年會專案評比第四名肯定。**我在此專案中除主導開發外，另一項關鍵貢獻是系統文件的撰寫與交付**——涵蓋演算法原理、API 介面、操作手冊等完整文件，確保後續工程師可順利接手維護。這段經驗讓我深刻理解：**程式碼本身不是交付物，可被組織持續使用的系統知識才是**。

**為什麼強化「文件主導」：**
- 對應雲科資管「**軟體品質管理**」「**知識管理**」課程
- 對應「**資訊技術與管理和組織的整合**」發展方向
- 比「我會寫程式」更貼近資管的訓練核心

## 六、針對「在校成績較弱」的補強策略

朋友建議「面試要著重比賽跟專題，**架構一定要懂**」——書審就要先**鋪好這條防線**。

### 策略：在每個專案內加入「架構決策」說明

評審看完書審如果只記得「他做過什麼」是不夠的，要讓他記得「**他為什麼這樣設計**」。

### 6-1 TWELF/FC 章節新增「架構決策思維」段落

```
▸ 架構決策思維

選擇 YOLOv7 + CNN-OCR 雙層架構而非端到端 OCR 的理由：
1. **問題分離**：試卷的「版面解析」（題目區、作答區、題號）
   與「文字辨識」（手寫 vs 印刷）本質是兩種任務，
   分層處理可獨立優化、獨立除錯
2. **資料效率**：YOLO 對於版面定位的標註成本遠低於
   端到端 OCR 的字元級標註
3. **可解釋性**：分層架構讓誤判可定位（是「框錯」還是「認錯」），
   對應教育場景的可信度需求

匹配演算法選擇歐氏距離而非 IoU 的理由：
試卷答案區的形狀變異性高（單行、多行、塗鴉），
IoU 對於「位置正確但形狀差異大」的情境容易誤判，
歐氏距離只看中心點，更貼合教育評分的實際語意。
```

### 6-2 DFI 章節新增「架構決策思維」段落

```
▸ 架構決策思維

選擇「卷積文字密度演算法」而非深度學習切割模型的理由：
1. **資料限制**：PCB 元件圖數量有限，深度學習易過擬合
2. **可解釋性**：傳產客戶需要的是「為什麼這樣切」的明確邏輯
3. **部署成本**：客戶端不必裝 GPU 或重型框架
4. **可調參數**：卷積核大小可由工程師根據新元件樣式直接微調

這次選型驗證了「**最先進的方案不等於最適合的方案**」——
這正是我希望在雲科資管「管理資訊系統」與「系統分析與設計」
中進一步系統化的判斷力。
```

### 這樣做的好處
- 面試官問「你印象最深的專題你怎麼解決」時，你可以**直接背架構決策**
- 評審讀書審就會覺得「**這個人不是只會調包**」
- 預先回答「資工 vs 資管」的問題（你已經在做資管的決策思維）

### Task 10 詳細說明：圖一時間軸學年標示修正

**現況問題：** 圖一時間軸目前以「年」為單位，但「專二 / 專三 / 專四 / 專五」的學期分界不夠精確，可能讓評審誤判你各段經歷的時間點與當下的年級身份。

**正確的學年分界（依台灣學制 9 月開學）：**

| 起始日 | 學年身份 |
|--------|----------|
| 2022/09 ~ 2023/08 | 專二 |
| 2023/09 ~ 2024/08 | 專三 |
| 2024/09 ~ 2025/08 | 專四 |
| 2025/09 ~ 2026/08 | 專五 |

**修改重點：**

1. **時間軸刻度切換點**：把學年分界線（垂直虛線或不同底色）放在每年的 **9 月**，而非 1 月或年中
2. **學年標籤位置**：每個學年區塊（9 月 → 隔年 8 月）正中央標示「專 X」
3. **各事件對齊驗證**：
   - 御先實習（2023/7-9）→ 跨在「專二暑假 → 專三開學」交界
   - 產學合作場地租借（2022/12-2023/3）→ 落在**專二上**
   - 模板設計 v1（2023/2-4）→ 落在**專二下**
   - iThome 鐵人賽（2023/10）→ 落在**專三上**
   - 開始籌備論文（2024/11）→ 落在**專四上**
   - TWELF（2025/4）→ 落在**專四下**
   - 樂聯（2025/6-9）→ 跨在「**專四暑假 → 專五開學**」
   - FC（2025/7）→ 落在**專四暑假**
   - DFI 友通 PCB（2025/6-12）→ 跨「**專四暑假 → 專五上**」
   - 專題競賽（2025/11-12）→ 落在**專五上**
   - v2 架站雲（2026/1-）→ 落在**專五下**
   - 工作坊講師（2026/6 預計）→ 落在**專五下尾聲**

**這個修正為什麼重要：**
- 雲科二技審查委員多為熟悉學制的教授，**時間軸精度錯誤會被看出來**
- 正確標示後可凸顯「**專四下到專五上是你的爆發期**」（論文 × DFI × 樂聯 × 專題競賽同時推進）
- 這段密集度本身就是「**抗壓性 / 多任務處理能力**」的視覺證明
